**Real bugs found and fixed in `internal/accounting`** (all reproduced live, not theoretical): (1) the consumer never retried a mid-batch DB failure once past initial startup — Redis `">"` never redelivers an already-delivered entry, so a Postgres outage hitting steady-state permanently stranded messages (`in_queue` never drained even after recovery) — fixed by forcing a retry via `id="0"`. (2) `consumerName()` was hostname+PID: any acct restart (crash, or a real host reboot) got a fresh Redis consumer identity and could never see the dead instance's own pending entries — fixed to a stable name (single-instance service, matches compose.yml). (3) `persisted`/`deduplicated`/`reaped` counters only survived a *graceful* shutdown; a SIGKILL right after boot (a real `unclean-reboot` scenario, not `docker compose up`'s `depends_on`-ordered start) reset them to whatever the last periodic flush wrote — fixed by committing them durably in the same DB transaction as the data, and mirroring `received`/`enqueued` into Redis. (4) that fix then raced the periodic flush (absolute `SET` vs. relative `+1` — a lost-update) — fixed by giving each counter exactly one writer. (5) `counters.load()` was single-shot; if Postgres was still mid-recovery at boot it zeroed *everything* — fixed with a bounded retry.
